## 1. 市場データ戦略の重大な欠落【最重要】

### 指摘された問題の確認

> 「入力が面倒、日本株・米国株の株価が正確に取得できない。日本株のAPIは公式のものが商用利用禁止されている」

**計画書の現状**: Section 6のアーキテクチャ再設計で以下のみ記載
```
yahoo-finance2 → 維持（無料）。収益確立後にClaude API追加
```

**これは不十分。** 市場データは本アプリの生命線であり、以下の問題がカバーされていない。

### 1.1 現在のデータソースの脆弱性分析

| ソース | 種類 | 問題 |
|--------|------|------|
| **yahoo-finance2 (npm)** | 非公式パッケージ | Yahoo Financeのスクレイピングラッパー。**いつでもブロック・破壊される可能性がある**。実際に過去何度もbreaking changeが発生。商用利用の保証なし |
| **JPX CSV** | 公式データ | 20分遅延。URL構造が変更されるリスク。日付指定URLのため取得失敗しやすい。銘柄名が日本語Shift-JIS |
| **Alpha Vantage** | 有料API | 無料枠: 25リクエスト/日。**日本株の対応が不完全**（東証全銘柄をカバーしていない） |
| **Webスクレイピング** | 非公式 | Minkabu/Kabutan/MarketWatch。**利用規約違反のリスク**。DOM構造変更で即死 |
| **フォールバック** | GitHub静的データ | 最終手段。データが古い。リアルタイム性ゼロ |

### 1.2 日本株APIの現実

| API | 料金 | 制限 |
|-----|------|------|
| **JPX Market Data Cloud** | 月額30万円〜 | 正式な商用API。個人開発者には非現実的 |
| **J-Quants API（JPX公式）** | 無料プランあり（2023年〜） | 無料: 12銘柄/リクエスト、1日制限あり。Standard: ¥1,650/月 |
| **kabuステーション API（auカブコム）** | 口座開設で無料 | リアルタイム。ただし個人利用限定、商用利用は要相談 |
| **楽天 RSS** | 口座開設で無料 | Excelアドイン。API化は非公式 |
| **Yahoo Finance (非公式)** | 無料 | スクレイピング。いつブロックされるか不明 |

### 1.3 改善提案: 市場データ戦略の追加

計画書にSection「市場データ戦略」を新設し、以下を記載すべき：

#### Stage 1（ゼロコスト期間）: 現実的な無料戦略
```
優先度1: J-Quants API 無料プラン（日本株）
  - JPX公式の合法的API
  - 無料プラン: 限定的だが商用利用可能
  - 日足データ + 銘柄一覧取得
  - 制限: 12銘柄/リクエスト、日次更新

優先度2: yahoo-finance2 npm（米国株・為替）
  - 米国株のカバレッジは良好
  - 為替レートも取得可能
  - 非公式であることを認識した上で使用

優先度3: JPX CSV（バックアップ）
  - J-Quants APIが落ちた場合のフォールバック

優先度4: ハードコードフォールバック
  - 全ソース失敗時の最終手段
```

#### Stage 2（収益確立後）: 有料API導入
```
日本株: J-Quants Standard（¥1,650/月）
  → 全銘柄対応、日中データ、財務データ

米国株: Financial Modeling Prep（$29/月）
  → リアルタイムデータ、バッチ対応、信頼性高い

為替: ExchangeRate-API（$9.99/月）
  → 信頼性高い為替データ
```

#### データ精度モニタリング（新規追加すべき）
```
- データソースごとの成功率・失敗率をCloudWatch Metricsで記録
- 日次でデータ精度レポートを自動生成
- 異常値検知（前日比±30%超の変動を警告）
- ソース切り替え時のデータ一貫性チェック
```

### 1.4 歴史的価格データの問題（「儲けが見えない」修正に必須）

計画書の問題2「儲けが見えない → 今日いくら増えた？」を実現するには、**購入時の価格データ**が必要。

**現状の問題:**
- ユーザーが入力した購入価格に依存
- 日次の価格推移データを保持していない
- 過去の価格データを取得するAPIが計画に含まれていない

**改善提案:**
```
1. 購入価格はユーザー入力 or CSVインポートで取得（現状維持）
2. 日次スナップショット: 毎日1回、保有銘柄の価格をDynamoDBに記録
   - テーブル: pfwise-price-history-{stage}
   - TTL: 2年間保持
   - Lambda scheduled function（1日1回）
3. ダッシュボードの損益計算:
   - 総損益 = Σ(現在価格 × 保有数 - 購入価格 × 保有数)
   - 前日比 = 今日のスナップショット - 昨日のスナップショット
   - 年初来 = 今日 - 1月1日のスナップショット
4. 初回は購入価格との差分のみ表示、履歴が蓄積されたら推移グラフを表示
```

## 2. アーキテクチャ移行計画の問題【重要】

### 2.1 Phase 0が「ビッグバンリライト」になっている

現在のPhase 0は7つのステップを一括で実行する構成：
```
Step 0-1: CRA → Vite
Step 0-2: 環境変数マイグレーション
Step 0-3: TypeScript導入
Step 0-4: require()除去
Step 0-5: Zustand移行
Step 0-6: Vitest移行
Step 0-7: クリーンアップ
```

**問題:**
- 全ステップを同時に進めると、何が壊れたか特定できない
- CRA→Vite移行だけでも数十ファイルに影響する
- 状態管理の移行（76KB PortfolioContext → Zustand）は最もリスクが高い
- テスト基盤も同時に変えると、移行中のデグレを検知できない

### 2.2 改善提案: Phase 0を3つのサブフェーズに分割

```
Phase 0-A: ビルド基盤（1〜2週間）
  ✅ Step 0-1: CRA → Vite
  ✅ Step 0-2: 環境変数マイグレーション
  ✅ Step 0-4: require() → import文
  ✅ Step 0-7: 不要ファイルクリーンアップ
  → 検証: 既存テストが全て通ること
  → デプロイ: 本番で動作確認

Phase 0-B: 型安全性（2〜3週間）
  ✅ Step 0-3: TypeScript導入（段階的）
  ✅ Step 0-6: Vitest移行
  → 検証: テストカバレッジが維持されていること
  → デプロイ: 本番で動作確認

Phase 0-C: 状態管理刷新（2〜3週間）
  ✅ Step 0-5: Zustand移行
  → 最もリスクが高いため最後に実行
  → PortfolioContext互換ラッパーで段階移行
  → 検証: 全機能の手動テスト + 自動テスト
  → デプロイ: 本番で動作確認
```

**各サブフェーズの完了条件:**
- 既存テストが全てパス
- 本番デプロイで主要機能が動作
- パフォーマンスが劣化していないこと

計画書にはフロントエンド・バックエンドの技術スタック変更は書かれているが、**データフローアーキテクチャ**の再設計が欠落。

**追加すべき内容:**

```
現状のデータフロー:
  ユーザー入力 → PortfolioContext（76KB） → localStorage
                                          → Google Drive（手動）
  市場データ → API → キャッシュ → PortfolioContext → 画面表示

新しいデータフロー（提案）:

    手動入力 → バリデーション → Zustand Store

  画面表示:
    Zustand Store + TanStack Queryキャッシュ → コンポーネント
```

## 3. デザイン方針の矛盾

### CLAUDE.md第8条との矛盾

**第8条**: 「デザインはhttps://atlassian.design/components を読み込み、これに準拠する」

**計画書**: 「shadcn/ui + Radix UI + TailwindCSS v4 + Framer Motion」

**矛盾点:**
- 既にAtlassianデザイントークンとコンポーネント（Button, Card, Input, Modal）が実装済み
- 計画書はshadcn/uiへの全面移行を提案
- 両方を維持するのは二重構造になる

**改善提案: 方針を明確に決定する**

```
選択肢A: Atlassian Design維持（CLAUDE.md準拠）
  - 既存実装を活用
  - エンタープライズ感を維持
  - ただしフィンテック向けデザインとしてはやや堅い

選択肢B: shadcn/ui移行（計画書の方針）
  - モダンなフィンテックUI
  - コミュニティが大きく、コンポーネントが豊富
  - CLAUDE.md第8条を更新する必要あり

選択肢C: ハイブリッド（推奨しない）
  - 二重構造になり、バンドルサイズ増大
  - デザインの一貫性が失われる
```

→ **決定を計画書に明記し、CLAUDE.md第8条も更新すべき**

## 6. 結論

**計画の方向性は正しいが、「市場データをどう正確に取得するか」という本アプリの根幹が不十分。**

現在のyahoo-finance2依存は**時限爆弾**であり、以下を最優先で実施すべき：

1. **J-Quants API（JPX公式）の調査とPoC** — 日本株の合法的なデータソース確保
2. **データ精度モニタリングの設計** — どのソースがどの程度正確かを常時監視
3. **歴史的価格データの蓄積設計** — 「儲けが見えない」問題の根本解決に必須

アーキテクチャ移行（Phase 0）は「ビッグバン」にならないよう、サブフェーズに分割し、各フェーズで本番デプロイ＋動作確認を挟むべき。
