# PFWise 長期計画品質レビュー

`documents/planning/redesign-plan.md` の計画内容をコードベース実測値と照合し、計画の精度・完全性・整合性を 5軸で評価します。

---

## 必須: プロジェクトドキュメント読み込み（最初に実行）

1. **`CLAUDE.md`（プロジェクトルート）** — コーディング原則・アーキテクチャ・技術スタック
2. **`documents/planning/redesign-plan.md`** — 計画本体（評価対象）
3. **`documents/review-guide.md`** — 5領域のレビュー基準

---

## STEP 1: コードベース実態計測（テスト実行込み）

以下のコマンドを実行して実測値を取得し、計画の数値と照合してください：

```bash
# 1. TypeScript エラー数（0 が理想）
cd frontend/webapp && npx tsc --noEmit 2>&1 | grep -c "error TS" || echo "0 errors"

# 2. テスト結果（pass/skip/fail 件数）
npx vitest run 2>&1 | tail -5

# 3. カバレッジ（statements/branches/functions/lines）
npx vitest run --coverage 2>&1 | grep "All files"

# 4. any 残存数（テスト除く）
grep -r ": any" src/ --include="*.ts" --include="*.tsx" | grep -v "__tests__" | wc -l

# 5. 絵文字残存（UI ファイルのみ）
grep -rn "🎯\|🔍\|💬\|⭐\|🔒\|📊\|💎\|🌐\|🏠\|🥇\|🥈\|🥉" src/ --include="*.tsx" | grep -v "__tests__" | wc -l

# 6. ハードコード色値残存（テスト除く）
grep -rn "#[0-9A-Fa-f]\{6\}\|#[0-9A-Fa-f]\{3\}" src/ --include="*.tsx" | grep -v "__tests__" | grep -v "OAuthLoginButton" | wc -l
```

実測値を「計画記載値 vs 実測値」の表にまとめてください。

## STEP 2: 完了フェーズの検証

`redesign-plan.md` で ✅ が付いた各フェーズについて以下を確認してください：

1. `git log --oneline -20` で対応コミットが存在するか確認
2. 計画の完了基準（acceptance criteria）が実際に満たされているか
3. 「完了」と記載されているが実態が「部分完了」の箇所を特定

**特に確認すべき点:**
- Phase 8-B: "4ストア移行完了" → 残タスクとして挙がっていた subscriptionStore 等の移行状況
- Phase 8-C: Zustand persist 統一 → 未着手のはずだが確認
- Phase 8-D: TypeScript strict: true → tsc --noEmit の結果で確認

---

## STEP 3: 5軸評価

### 軸1. 完了精度 【配点：20点】

**確認項目:**
- [ ] ✅ 付きフェーズが実際にコードベースで完了している
- [ ] 完了基準（数値目標）が定量的で検証可能
- [ ] 「部分完了」が✅になっていない（中途半端な完了なし）
- [ ] 完了日・コミットハッシュが記録されている
- [ ] 計画記載のメトリクスと実測値が一致している（±5%以内）

| 点数 | 基準 |
|------|------|
| 18〜20 | 全完了フェーズが実測で確認、メトリクスも正確 |
| 14〜17 | おおむね正確だが一部に軽微な乖離がある |
| 10〜13 | 複数フェーズで完了状態に乖離がある |
| 0〜9 | 計画と実態が大きくかけ離れている |

---

### 軸2. 次フェーズ明確性 【配点：20点】

**確認項目:**
- [ ] 「次回実行」フェーズの作業内容が具体的（ファイル名・関数名レベル）
- [ ] 受け入れ基準（acceptance criteria）が定量的
- [ ] フェーズ間の依存関係が正しく記述されている
- [ ] 見積もり工数・優先度が示されている
- [ ] 各フェーズが1〜2週間以内に完了可能なスコープか

| 点数 | 基準 |
|------|------|
| 18〜20 | 次フェーズが詳細・定量的、依存関係も明確 |
| 14〜17 | 概要は明確だが実装詳細や工数見積もりが不足 |
| 10〜13 | 「何をするか」は書かれているが「どうするか」が不明 |
| 0〜9 | 次のアクションが不明確で実行計画として機能しない |

---

### 軸3. メトリクス正確性 【配点：20点】

**確認項目（STEP 1の実測値と比較）:**
- [ ] テスト合格数の記載値 vs 実測値（±10件以内）
- [ ] テストカバレッジ率の記載値 vs 実測値（±2%以内）
- [ ] any 残存数の記載値 vs 実測値（目標値が現実的か）
- [ ] 絵文字・ハードコード色値の記載値が実測と一致
- [ ] 成功指標テーブルの「現在」列が最新状態を反映している

| 点数 | 基準 |
|------|------|
| 18〜20 | 全メトリクスが実測値と一致（計画がダッシュボードとして機能） |
| 14〜17 | ほぼ正確、一部が古い値のまま |
| 10〜13 | 複数のメトリクスが古い・不正確 |
| 0〜9 | メトリクスの多くが実態と乖離している |

---

### 軸4. 技術的整合性 【配点：20点】

**確認項目:**
- [ ] Phase 間の依存関係が CLAUDE.md のアーキテクチャと矛盾しない
- [ ] 技術的負債（8-C, 8-B残タスク）が計画に正しく反映されている
- [ ] TypeScript strict: true との整合性（新フェーズで型安全性を維持できるか）
- [ ] TanStack Query / Zustand パターンが次フェーズでも継続される設計か
- [ ] テスト戦略（Vitest + MSW）が次フェーズの実装に対応できるか

| 点数 | 基準 |
|------|------|
| 18〜20 | アーキテクチャ整合性が高く、技術負債も計画済み |
| 14〜17 | 大部分は整合、一部の技術負債が計画に未反映 |
| 10〜13 | 複数の技術的リスクが計画に反映されていない |
| 0〜9 | アーキテクチャと計画が矛盾している |

---

### 軸5. ビジネス整合性 【配点：20点】

**確認項目（タケシのペイン + Free/Standard モデル）:**
- [ ] 次フェーズがタケシの3大ペイン（複数口座/リバランス/投資不安）のいずれかを解消する
- [ ] Standard（¥700/月）の価値向上に直接貢献するフェーズが計画されている
- [ ] Free→Standard の自然なアップグレード導線が設計に含まれている
- [ ] ロードマップが3〜6ヶ月の現実的な時間軸で描かれている
- [ ] KPI（テスト/カバレッジ/E2E数）の目標値がビジネス目標と連動している

| 点数 | 基準 |
|------|------|
| 18〜20 | タケシのペイン解消と収益化が次フェーズに直結している |
| 14〜17 | ビジネス価値はあるが収益化導線の設計が不足 |
| 10〜13 | 技術改善中心で収益化への貢献が見えにくい |
| 0〜9 | ビジネス目標と計画が乖離している |

## レビュー後の作業

- **計画更新が必要な場合**: `documents/planning/redesign-plan.md` を直接修正する
- **クリティカルな乖離がある場合**: `documents/tmp/plan-review-{日付}.md` に詳細報告書を作成
- **パターン化した問題**: `documents/review-guide.md` の「計画品質」セクションに追記
