# PFWise テストカバレッジギャップ分析

ソースファイルとテストファイルを突き合わせ、テスト未実装・テスト品質不足のファイルを特定し、優先度付きギャップ一覧を出力します。

---

## 必須: プロジェクトドキュメント読み込み（最初に実行）

1. **`CLAUDE.md`（プロジェクトルート）** — テストアーキテクチャ・アンチパターン定義
2. **`documents/review-guide.md`** — テスト品質レビュー基準
3. **`MEMORY.md`（.claude/）** — テスト品質アンチパターン一覧

---

## STEP 1: ソースファイル一覧の収集

`frontend/webapp/src/` 配下の以下カテゴリのファイルを全て列挙してください：

| カテゴリ | パスパターン | 優先度 |
|---------|------------|--------|
| ページ | `src/pages/**/*.{ts,tsx}` | P0 |
| ストア | `src/stores/**/*.{ts,tsx}` | P0 |
| サービス | `src/services/**/*.{ts,tsx}` | P0 |
| ユーティリティ | `src/utils/**/*.{ts,tsx}` | P0 |
| フック | `src/hooks/**/*.{ts,tsx}` | P1 |
| コンポーネント | `src/components/**/*.{ts,tsx}` | P1 |

**除外**: `index.ts`（再エクスポートのみ）、`*.d.ts`（型定義のみ）、`src/components/ui/`（shadcn/ui プリミティブ）

---

## STEP 2: テストファイルとのクロスリファレンス

`__tests__/` ディレクトリ配下のテストファイルを全て列挙し、各ソースファイルに対応するテストの有無を判定してください。

**対応ルール:**
- `src/stores/portfolioStore.ts` → `__tests__/unit/stores/portfolioStore.test.ts`
- `src/components/dashboard/Foo.tsx` → `__tests__/unit/components/dashboard/Foo.test.tsx`
- `src/utils/bar.ts` → `__tests__/unit/utils/bar.test.ts`
- ファイル名・パス構造で一致するものを対応テストとみなす

---

## STEP 3: テスト未実装ファイルの特定

STEP 2 で対応テストが見つからなかったファイルを「テスト未実装」として記録してください。

---

## STEP 4: テスト品質の評価

テストが存在するファイルについて、以下の基準で品質グレードを付与してください。

### グレード基準

| グレード | 基準 |
|---------|------|
| **A（優良）** | 正常系＋異常系＋境界値をカバー。振る舞いテスト中心。具体値でアサーション |
| **B（標準）** | 正常系はカバー。一部異常系テストあり。アサーションはおおむね具体的 |
| **C（浅い）** | render チェックのみ、または `toBeDefined()` / `toMatchSnapshot()` のみ。振る舞いテストなし |
| **F（欠陥）** | テストが存在するが壊れている、スキップされている、または意味のないアサーションのみ |
| **-（未実装）** | テストファイルが存在しない |

### 浅いテストの検出パターン

以下のパターンが**テストの大半を占める**場合、グレードを C 以下に下げてください：

```
# 弱いアサーション（値の検証をしていない）
expect(X).toBeDefined()
expect(X).toBeTruthy()
expect(X).toBeGreaterThan(0)     # 金融計算で具体値を検証していない
expect(X).not.toBeNull()

# レンダリングのみ（振る舞い未検証）
render(<Component />)
expect(screen.getByText('...')).toBeInTheDocument()
# ↑ これだけでユーザー操作（click, input）テストなし

# スナップショット依存
expect(X).toMatchSnapshot()
expect(X).toMatchInlineSnapshot()
```

## STEP 5: 通貨変換テストの専用チェック

以下の条件に該当するソースファイルに対し、通貨変換テストの有無を確認してください：

**対象条件**（ソースファイルに以下のパターンが含まれる）:
- `price * holdings` または `price * quantity` の計算
- `exchangeRate` の参照
- `currency` による分岐（`if (currency === 'JPY')` 等）
- `convertCurrency` / `currencyUtils` のインポート

**チェック項目:**
- [ ] JPY 資産のみのテストケースがある
- [ ] USD 資産のみのテストケースがある
- [ ] JPY + USD 混在ポートフォリオのテストケースがある
- [ ] `baseCurrency` が JPY の場合と USD の場合の両方をテスト
- [ ] `exchangeRate` が `0` / `undefined` / `NaN` の異常系テストがある
- [ ] `currency: undefined`（通貨未設定）のエッジケースがある

---

## STEP 6: 不足シナリオの特定

各ファイルについて、テストで**カバーされていない重要シナリオ**を特定してください：

**ストア（Zustand）の必須シナリオ:**
- 初期状態 → アクション実行 → 状態変化の検証
- 複数アクションの連鎖（例: addAsset → calculateTotal → updateScore）
- persist ミドルウェアの復元テスト

**サービス層の必須シナリオ:**
- API 成功レスポンス
- 401 → トークンリフレッシュ → リトライ
- 5xx → フォールバックデータ
- ネットワークエラー

**コンポーネントの必須シナリオ:**
- ユーザー操作（click, input, select）
- ローディング状態の表示
- エラー状態の表示
- Free/Standard プラン分岐の表示

## 注意事項

- ファイルの内容を読む際は Serena MCP を活用して効率的にシンボル構造を把握すること
- テストファイルのアサーション品質は実際のテストコードを読んで判断すること（ファイル名だけで判断しない）
- 通貨変換チェックは MEMORY.md に記載されたバグ修正パターンと照合すること

ARGUMENTS: （特定のカテゴリやファイルパスを指定可能。省略時は全カテゴリを分析）
