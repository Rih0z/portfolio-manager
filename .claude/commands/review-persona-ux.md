# PFWise ペルソナUX適合レビュー（4軸・100点満点）

ペルソナ「タケシ」の視点でUI/UXを総合レビューし、「毎日開きたくなるアプリ」かどうかを評価します。

---

## 必須: プロジェクトドキュメント読み込み（最初に実行）

1. **`CLAUDE.md`** — コーディング原則・アーキテクチャ・技術スタック
2. **`documents/tmp/redesign-plan.md`** — ペルソナ定義（タケシ）・収益モデル
3. **`documents/tmp/competitor-analysis-dopamine-design.md`** — 競合分析・ドーパミン設計原則
4. **`documents/tmp/phase-dopamine-ux-plan.md`** — ダッシュボード再設計計画
5. **`documents/review-guide.md`** — レビュー基準
6. **`documents/ui-review-checklist.md`** — デザインチェックリスト

---

## STEP 0: レビュー対象の特定

引数が指定されている場合：その機能・ファイル・コンポーネントをレビュー対象とする。
引数がない場合：ダッシュボード全体（`src/pages/Dashboard.tsx` + 関連コンポーネント）をレビュー対象とする。

---

## STEP 1: ペルソナ「タケシ」プロファイル確認

`documents/tmp/redesign-plan.md` から以下を確認：

- **年齢**: 28〜42歳、IT企業勤務（エンジニアまたはPM）
- **投資歴**: 2〜7年、運用額500万〜5,000万円
- **使用証券**: SBI証券 + 楽天証券（+ マネックス証券）
- **デバイス**: 主にiPhone（通勤中にチェック）、自宅ではPC
- **性格**: データ重視、自分で判断したい、無責任なゲーミフィケーションを嫌う
- **3大ペイン**: ①複数口座統合 ②リバランス計算 ③投資判断の不安

---

## STEP 2: コードベース調査

Serena MCP を使って対象コンポーネントを調査：

1. `get_symbols_overview` でダッシュボード関連ファイルのシンボル一覧を取得
2. `find_symbol` で主要コンポーネントのbodyを読む
3. 情報階層（コンポーネント配置順序）を確認
4. レスポンシブ対応（モバイル表示）を確認

---

## STEP 3: 4軸評価

### 軸1. ドーパミン設計チェック 【配点：25点】

**タケシが「開いてよかった」と3秒で感じるか？**

| # | チェック項目 | 配点 | 確認方法 |
|---|-------------|------|---------|
| 1-1 | 開いた瞬間に「評価額」が最大フォントで見えるか | 5 | Dashboard.tsx の最初のコンポーネントを確認 |
| 1-2 | 利益/損失が色（緑/赤）で直感的にわかるか | 5 | PnL表示コンポーネントのclassName/style確認 |
| 1-3 | 期間別比較（前日/先週/先月）で成長実感があるか | 5 | 期間切替UI・計算ロジックの有無 |
| 1-4 | スコア変動（前回比）が可視化されているか | 5 | PortfolioScoreCard のバッジ表示確認 |
| 1-5 | 「毎日開きたくなる理由」が1つ以上あるか | 5 | ウィークリーダイジェスト・通知・新情報の有無 |

**競合比較基準:**
- Robinhood: 金額ドカン + 全画面色フィードバック → 満点基準
- ロボフォリオ: 総資産+前日比が最上部 → 15点基準
- 現状PFWise（改善前）: 情報が埋もれている → 5〜8点相当

| 点数 | 基準 |
|------|------|
| 22〜25 | 開いた瞬間に投資成果が飛び込む、色・サイズ・位置全て最適 |
| 17〜21 | 成果は見えるが、インパクトや視覚フィードバックが不足 |
| 10〜16 | 情報はあるがスクロールが必要、または視覚的に埋もれている |
| 0〜9 | 投資成果がファーストビューに存在しない |

### 軸2. ペルソナ適合チェック 【配点：25点】

**タケシの実際の使い方に合っているか？**

| # | チェック項目 | 配点 | 確認方法 |
|---|-------------|------|---------|
| 2-1 | 語彙がIT系30代に適切か（専門用語OK、過度な簡略化NG） | 5 | UI文言のレビュー |
| 2-2 | モバイルファーストか（通勤中にiPhoneで5秒チェック） | 5 | レスポンシブCSS、タッチターゲット44px |
| 2-3 | 情報密度は適切か（データ重視のIT系に薄すぎないか） | 5 | ファーストビューの情報量 |
| 2-4 | CSV取込→ダッシュボード反映の最短パス（Aha moment） | 5 | Import→反映までのステップ数 |
| 2-5 | フリクションポイントがないか（不要な確認・過剰入力） | 5 | 操作フロー全体のステップ確認 |

**タケシの1日の使い方シミュレーション:**
1. 朝8:00 通勤電車 → アプリ開く → 「今日の評価額」確認（5秒）
2. → スコア確認「78点、先週から+3」→ 安心（3秒）
3. → 気になる銘柄の変動チェック（10秒）
4. → 閉じる（合計20秒以内）

このフローが自然に成立するか？

| 点数 | 基準 |
|------|------|
| 22〜25 | タケシの20秒チェックが完璧に成立、Aha momentも最短 |
| 17〜21 | おおむね適合だが一部でスクロールや操作が多い |
| 10〜16 | 基本機能はあるがタケシの使い方との乖離がある |
| 0〜9 | ペルソナ分析がUIに反映されていない |

---

### 軸3. UIデザインチェック 【配点：25点】

**フィンテック信頼感 + shadcn/ui準拠**

| # | チェック項目 | 配点 | 確認方法 |
|---|-------------|------|---------|
| 3-1 | shadcn/ui コンポーネント使用（独自実装禁止） | 5 | import元確認 |
| 3-2 | フィンテック信頼感カラー（Blue/Grey基調、控えめ配色） | 5 | CSS変数・TailwindCSS確認 |
| 3-3 | JetBrains Mono + tabular-nums で金額表示 | 5 | font-family・className確認 |
| 3-4 | 緑赤は損益のみに限定（他の意味で使わない） | 5 | 全コンポーネントの色使い確認 |
| 3-5 | 絵文字不使用（Lucide Icons のみ） | 5 | JSX内の絵文字検索 |

**フィンテック信頼感の定義（タケシ向け）:**
- 「銀行アプリのような信頼感」 + 「モダンテック企業の洗練さ」
- NOT: ゲームアプリ、SNSアプリ、カジュアルアプリ
- 参考: WealthNavi の落ち着いた色調 + Robinhood の情報階層

| 点数 | 基準 |
|------|------|
| 22〜25 | shadcn/ui完全準拠、フィンテック信頼感、表示統一 |
| 17〜21 | おおむね準拠だが一部で独自スタイルや不統一がある |
| 10〜16 | 基本的なデザインはあるが統一感に欠ける |
| 0〜9 | デザインシステムが未適用、絵文字使用、信頼感なし |

---

### 軸4. エンゲージメントチェック 【配点：25点】

**「来てよかった」→「明日も来よう」のサイクルが成立するか？**

| # | チェック項目 | 配点 | 確認方法 |
|---|-------------|------|---------|
| 4-1 | 「来てよかった」が3秒以内にあるか（即時の価値提供） | 5 | ファーストビューの即時性 |
| 4-2 | Aha moment（初回CSV取込→ダッシュボード反映）が最短か | 5 | 初回ユーザーフロー確認 |
| 4-3 | Free→Standard の転換トリガーが自然か | 5 | UpgradePrompt配置・タイミング |
| 4-4 | 情報階層が明確か（最重要→補助の順） | 5 | Dashboard.tsx のコンポーネント順序 |
| 4-5 | 空状態が「行動を促す」設計か（CSV未取込時） | 5 | 空状態コンポーネント確認 |

**エンゲージメントループ設計:**
```
開く → 評価額確認（3秒）→ 変動確認（5秒）→ 「順調だ」→ 閉じる → 翌日また開く
         ↑                                                        ↓
    マイルストーン達成！ ←───── 資産増加 ←───── 良い投資行動 ←──┘
```

このループが機能しているか？

| 点数 | 基準 |
|------|------|
| 22〜25 | エンゲージメントループが完全に成立、翌日の再訪動機がある |
| 17〜21 | 基本ループは成立するが再訪動機が弱い |
| 10〜16 | 機能は使えるが「また来たい」理由がない |
| 0〜9 | 使い切り体験、リテンション設計がない |
