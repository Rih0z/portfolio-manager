## 1. テスト品質レビュー

### 1-A. テストが検証すべき対象（優先順）

| 優先度 | 対象 | 理由 |
|--------|------|------|
| **P0** | 金融計算ロジック | PnL・為替変換・ポートフォリオスコアの誤りは信頼喪失に直結 |
| **P0** | 認証・セッション管理 | JWT/Session二重認証の不整合はデータ漏洩リスク |
| **P1** | サブスクリプション制限 | Free/Standard機能制限の抜け穴は収益損失 |
| **P1** | データ永続化 | Google Drive同期・localStorage・サーバー保存の一貫性 |
| **P2** | ユーザーフロー | CSV取込→ダッシュボード表示→シミュレーション→AI分析 |
| **P2** | エラーハンドリング | API障害・ネットワーク断・データ不整合時のフォールバック |

### 1-B. テスト品質チェックリスト

#### 構造的品質
- [ ] **ビジネスロジックテスト**: ユーティリティ・サービス・ストアの各関数に対して、入力→期待出力を検証するテストが存在するか
- [ ] **境界値テスト**: 0件・1件・上限値（Free: 5銘柄）・負値・null/undefinedのケースをカバー
- [ ] **異常系テスト**: API 401/403/500応答、ネットワークタイムアウト、不正データ形式に対する処理を検証
- [ ] **状態遷移テスト**: Zustandストアの状態変更が正しい順序で発生するか（loading→success/error）

#### アンチパターン検出
- [ ] **スナップショット依存**: `toMatchSnapshot()` に頼っていないか（振る舞いテストを優先）
- [ ] **実装詳細テスト**: 内部メソッド名・CSS クラス名への依存がないか
- [ ] **テスト内ハードコード**: マジックナンバーや固定日付が放置されていないか
- [ ] **過剰モック**: 本来テスト対象であるべきモジュールまでモックしていないか
- [ ] **アサーション不足**: `toBeDefined()` のみで具体的な値を検証していないか
- [ ] **テスト間依存**: テスト実行順序に依存する共有状態がないか（`beforeEach` で必ずリセット）

#### カバレッジ基準
- [ ] **閾値遵守**: statements ≥ 75%, lines ≥ 75%, functions ≥ 70%, branches ≥ 65%
- [ ] **新規コード**: 追加したコードのカバレッジが80%以上
- [ ] **未テスト除外の妥当性**: `vitest.config.ts` の `coveragePathIgnorePatterns` に正当な理由がある

### 1-C. テスト種別と期待品質

| テスト種別 | 対象ディレクトリ | 期待品質 |
|-----------|----------------|---------|
| **ユニットテスト** | `src/__tests__/unit/` | 純粋関数の入出力検証。外部依存は `vi.mock()` で隔離 |
| **ストアテスト** | `src/__tests__/unit/stores/` | `getState().action()` → 状態変化の検証。クロスストア通信はモック |
| **サービステスト** | `src/__tests__/unit/services/` | API呼び出しパターン・エラーハンドリング・リトライロジック |
| **コンポーネントテスト** | `src/__tests__/unit/components/` | React Testing Library でユーザー操作→画面変化を検証 |
| **E2Eテスト** | `__tests__/e2e/` | Playwright でクリティカルパス（ログイン→CSV取込→ダッシュボード→エクスポート） |

### 1-D. レビュー実行手順

```
1. Serena MCP でテスト対象のシンボルを特定
   → find_symbol でテスト対象の関数/クラスを確認
   → find_referencing_symbols でテストファイルを特定

2. テストファイルの構造を確認
   → get_symbols_overview でテストケース一覧を取得
   → describe/it のネスト構造が論理的か確認

3. テスト品質を評価
   → 上記チェックリストに照らして個別テストを読む
   → アンチパターンの有無を確認

4. カバレッジを確認
   → npm run test:coverage で全体カバレッジ
   → 対象ファイルの行カバレッジを確認

5. 問題があれば docs/tmp/ に報告
```

## 2. 処理正確性レビュー

### 2-A. 金融計算の正確性

| 計算 | 場所 | 検証ポイント |
|------|------|-------------|
| **PnL（損益）** | `utils/plCalculation.ts` | 取得単価×数量 vs 現在価格×数量。通貨混在時の為替適用順序 |
| **為替変換** | `utils/currencyUtils.ts` | JPY↔USD変換の丸め方向（銀行丸め）。レート未取得時のフォールバック値 |
| **ポートフォリオスコア** | `utils/portfolioScore.ts` | 8指標の計算式が redesign-plan.md の定義と一致。Free(3指標)/Standard(8指標)の分岐 |
| **リバランス** | `utils/rebalanceCalculation.ts` | 目標配分との差分計算。売買数量の切り捨て/切り上げ方向 |
| **目標達成率** | `utils/goalCalculations.ts` | 複利計算・積立シミュレーション。月次/年次の期間計算 |

#### チェックリスト
- [ ] **通貨の一貫性**: 全計算で通貨単位が明示され、混合計算時に為替変換が適用されている
- [ ] **参考値表示**: yahoo-finance2 由来のデータに `※参考値` ディスクレーマーが付与されている
- [ ] **端数処理**: 金額表示で適切な小数点以下桁数（JPY: 0桁、USD: 2桁）
- [ ] **ゼロ除算防止**: 配分比率0%・保有数量0・為替レート0の場合にNaN/Infinityが発生しない
- [ ] **日付境界**: 月末・年末・閏年でのPnL期間計算が正しい

### 2-B. 状態管理の正確性

#### Zustandストア整合性
- [ ] **authStore**: ログイン→JWT保存→リフレッシュ→ログアウトの状態遷移が漏れなくカバー
- [ ] **portfolioStore**: アセット追加/削除後にサマリー（合計額・配分比率）が再計算される
- [ ] **subscriptionStore**: プラン変更後に `isPremium()` と `canUseFeature()` が即座に反映
- [ ] **uiStore**: テーマ/言語変更がグローバルに適用され、localStorage に永続化

#### クロスストア整合性
- [ ] **認証→データ**: ログアウト時に portfolioStore のデータがクリアされる
- [ ] **サブスク→機能制限**: プランダウングレード時に超過データの扱いが定義されている
- [ ] **ポートフォリオ→目標**: アセット変更時に目標達成率が再計算される

### 2-C. API連携の正確性

- [ ] **リクエスト形式**: バックエンドAPIスペック（`docs/api-specification.md`）と一致
- [ ] **認証ヘッダー**: JWT Bearer Token が全APIリクエストに付与（`withCredentials: true`）
- [ ] **エラーレスポンス**: 401→トークンリフレッシュ→リトライ、403→UpgradePrompt、5xx→graceful degradation
- [ ] **レート制限**: Free(30/h) / Standard(300/h) の制限に対するUI側のフィードバック
- [ ] **キャッシュ整合性**: TanStack Query のキャッシュキーが一意で、staleTime が適切

### 2-D. レビュー実行手順

```
1. Serena MCP で対象処理のシンボルを読む
   → find_symbol で関数のbodyを取得
   → 計算ロジックの正確性を数学的に検証

2. データフローを追跡
   → find_referencing_symbols で呼び出し元を確認
   → 入力データの型・範囲を確認

3. エッジケースを列挙
   → 空配列、null、0、負値、極大値
   → 通貨混在、為替レート未取得、API障害

4. テストとの対応を確認
   → 各エッジケースに対応するテストが存在するか

5. 問題があれば docs/tmp/ に報告
```

## 3. マネタイズ整合性レビュー

### 3-A. 収益モデル定義（参照: redesign-plan.md）

| 項目 | Free | Standard (¥700/月 or ¥7,000/年) |
|------|------|--------------------------------|
| 保有銘柄数 | 5 | 無制限 |
| 市場データ取得 | 3回/日（参考値） | リアルタイム（参考値） |
| リバランスシミュレーション | 3回/月 | 無制限 |
| ポートフォリオスコア | 基本（3指標） | 詳細（8指標） |
| AIプロンプト | 1回/月 | 無制限 |
| PnLトラッキング | 合計のみ | 日次/YTD/トレンドグラフ |
| 目標設定 | 1件 | 5件 |
| Driveバックアップ | 手動 | 自動（日次） |
| データエクスポート | CSV | CSV + JSON + PDF |
| 広告 | あり（最小限） | なし |

### 3-B. チェックリスト

#### 機能制限の実装
- [ ] **ゲーティング実装**: 上表の全機能に対して `subscriptionStore.canUseFeature()` または `isPremium()` による制限が実装されている
- [ ] **制限到達時UX**: 制限に達した時に `UpgradePrompt` が表示され、`/pricing` へ導線がある
- [ ] **制限の正確性**: Free ユーザーが Standard 機能にアクセスできる抜け穴がない
- [ ] **使用量カウント**: シミュレーション回数・AIプロンプト回数が正しくカウントされ、月次リセットされる

#### 収益フローの完全性
- [ ] **Checkout フロー**: `/pricing` → `startCheckout()` → Stripe Checkout → 成功/キャンセルの全パス
- [ ] **プラン管理**: 既存加入者が `openPortal()` でStripe Customer Portal にアクセス可能
- [ ] **ステータス同期**: `fetchStatus()` が認証後・チェックアウト後・アプリ復帰時に呼ばれる
- [ ] **年額割引表示**: ¥7,000/年 (¥1,400お得) の表示が正確
- [ ] **解約後の扱い**: サブスク解約後にFreeプラン機能に正しくフォールバック

#### 価値訴求の一貫性
- [ ] **Free→Standard 導線**: ユーザーがFree機能を使い、自然に制限を感じてStandardへ導かれる設計
- [ ] **UpgradePrompt の配置**: 制限到達時のみ表示（過度な表示はUX悪化）
- [ ] **Pricing ページ**: 機能比較表が最新のプラン定義と一致
- [ ] **LP のプラン表示**: Landing ページの料金セクションが Pricing ページと一致

#### アナリティクス
- [ ] **コンバージョンファネル追跡**: `landing_view` → `pricing_view` → `checkout_start` → `checkout_complete` の各イベントが送信
- [ ] **プラン別行動分析**: Free/Standard ユーザーの機能利用頻度が識別可能

### 3-C. レビュー実行手順

```
1. redesign-plan.md のプラン定義を確認
   → docs/planning/redesign-plan.md の「料金プラン」セクション

2. subscriptionStore の機能制限を検証
   → find_symbol で canUseFeature / isPremium を読む
   → 全制限項目が実装されているか照合

3. UIでの制限表示を確認
   → UpgradePrompt の使用箇所を find_referencing_symbols で特定
   → 各制限ポイントで適切なフィードバックがあるか

4. Stripe連携の完全性
   → subscriptionService.ts のエンドポイントを確認
   → チェックアウト→成功→ステータス更新のフロー

5. Pricing ページとLP の表示一致を確認

6. 問題があれば docs/tmp/ に報告
```

## 5. ペルソナ整合性レビュー

### 5-A. ターゲットペルソナ定義（参照: redesign-plan.md）

#### プライマリ: 「テック系長期投資家 タケシ」
| 属性 | 詳細 |
|------|------|
| **年齢** | 28〜42歳 |
| **職業** | IT企業（エンジニア・PM・デザイナー）/ スタートアップ |
| **年収** | 600万〜1,200万円 |
| **投資歴** | 2〜7年、インデックス中心、長期分散、月5万〜30万円積立 |
| **投資額** | 500万〜5,000万円（運用中） |
| **利用サービス** | SBI/楽天証券、MoneyForward（不満あり）、X投資情報 |
| **ペイン** | ①複数口座の統合管理 ②リバランス計算が面倒 ③投資判断の不安 |
| **ゲイン** | ①全体像を一目で把握 ②最適リバランス自動化 ③AIに相談 |

#### セカンダリ: 「慎重な投資初心者 ユキ」（Phase 2+）
| 属性 | 詳細 |
|------|------|
| **年齢** | 25〜35歳 |
| **投資歴** | 0〜2年（新NISA開始組） |
| **ペイン** | 何を買えばいい？損失が怖い |
| **ゲイン** | 資産の可視化、少額でも成長実感 |

### 5-B. ペルソナ整合性チェックリスト

#### 課題解決の検証（タケシのペイン→ゲイン）

**ペイン①: 複数口座の統合管理**
- [ ] CSVインポートがSBI・楽天・汎用フォーマットに対応し、Shift-JIS自動検出が機能する
- [ ] 複数回のCSV取込でデータが統合（上書きではなく追加/更新）される
- [ ] ダッシュボードで全保有資産が一元表示され、通貨別集計ができる
- [ ] 「タケシ」が日常使うSBI/楽天の操作フローに近い体験になっている

**ペイン②: リバランス計算が面倒**
- [ ] 目標配分を設定→現状との差分→売買提案が3ステップ以内で完了する
- [ ] シミュレーション結果が具体的な売買金額/数量で表示される
- [ ] 結果をエクスポートして証券口座での発注に使える
- [ ] リバランス頻度の推奨（月次/四半期）が表示される

**ペイン③: 投資判断の不安**
- [ ] ポートフォリオスコア（100点満点）が現状の問題点を明確に示す
- [ ] AIプロンプトが具体的なアクションを含む（曖昧な一般論ではない）
- [ ] 3つの分析視点（リスク・コスト・成長性）がバランスよく表示される
- [ ] 「タケシ」が信頼できるデータソース表示（※参考値の明示）

#### UX フロー検証

**オンボーディング（初回利用）**
- [ ] LP → 30秒デモ/説明 → ログイン → CSV取込 OR テンプレート → ダッシュボード の流れが3分以内で完了可能
- [ ] 初回セットアップウィザードが通貨・予算・市場選択を簡潔にガイド
- [ ] CSV未取込でもテンプレートデータでダッシュボードを体験可能

**日常利用（継続価値）**
- [ ] ダッシュボード表示まで3タップ/クリック以内（アプリ起動→認証→表示）
- [ ] PnL（損益）が前日比で即座に確認できる（タケシの「朝の確認」ユースケース）
- [ ] 月次投資レポートが自動生成され、投資記録として蓄積される
- [ ] 通知（価格アラート・目標達成・リバランス提案）が過度でなく有用

**アップグレード動機**
- [ ] Free で十分な価値を感じた後に、自然にStandard の必要性を認識する設計
- [ ] 「タケシ」の投資規模（500万〜5,000万円）に対して¥700/月が妥当に感じられる価値提供
- [ ] 5銘柄制限が「タケシ」の典型的な保有数（10-30銘柄）に対して明確な制約として機能

#### テクノロジー親和性

- [ ] **モダンUI**: 「タケシ」がIT従事者として期待するモダンなインターフェース
- [ ] **レスポンス速度**: ダッシュボード表示・計算結果が2秒以内（タケシは遅いUIに不満）
- [ ] **データ透明性**: 計算根拠・データソースが確認可能（エンジニアとしての検証欲求）
- [ ] **PWA対応**: スマホのホーム画面に追加して即アクセス（アプリ的体験）
- [ ] **キーボード操作**: デスクトップでの効率的な操作（Tabキー・ショートカット）

#### 言語・文化的整合性

- [ ] **日本語自然さ**: 翻訳調ではなく、日本人が書いた自然な日本語
- [ ] **金融用語**: 一般的な投資用語（リバランス、ポートフォリオ、分散投資）を使用。専門的すぎない
- [ ] **金額表示**: 日本円は `¥1,234,567`（3桁区切り）、米ドルは `$1,234.56`
- [ ] **日付形式**: `2026年3月8日` または `2026/03/08`（ISO形式はUI非表示）

### 5-C. レビュー実行手順
